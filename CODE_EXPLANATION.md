# Trade Lifecycle Management System: Code Explanation

This document explains the project as if you built it and need to defend every technical decision. It focuses on the human-written project files. Generated files like `package-lock.json` and binary screenshot files are explained by purpose because they are not application logic.

## How The App Works Overall

The project is a full-stack Node.js and PostgreSQL application with a plain HTML/CSS/JavaScript frontend.

The frontend in `public/` shows the dashboard, forms, market overview, latest trades, trade detail modal, and audit trail.

The backend in `src/` exposes REST APIs through Express. The route files define endpoint URLs. The controller files handle request/response logic. The service files contain reusable business logic such as validation, P&L calculation, market data fetching, caching, and audit logging.

PostgreSQL stores instruments, trades, and audit logs. The `database.sql` file creates the database tables and seed data.

### `src/controllers/marketDataController.js`

Imports:

- `pool`: checks instrument validity.
- `getLatestMarketPrice`: fetches/cache market prices.
- `createAuditLog`: records a market price refresh event.

`isActiveInstrument(symbol)` checks PostgreSQL to ensure the symbol exists and is active.

`getMarketPrice(req, res)`:

1. Reads `symbol` from `req.params`.
2. Trims and uppercases it.
3. Validates it against `instruments`.
4. Rejects invalid/inactive symbols with HTTP 400.
5. Calls `getLatestMarketPrice(symbol)`.
6. Writes a `MARKET_PRICE_REFRESHED` audit event.
7. Returns market price, source, quote timestamp, checked time, cache age, cache flag, and stale flag.
8. Returns HTTP 502 if the provider/cache cannot provide a price.

The important design point: frontend never calls Twelve Data directly. The backend protects the API key and controls caching.

### `src/controllers/marketOverviewController.js`

Imports:

- `pool`: loads instruments.
- `getLatestMarketPrice`: gets market prices with cache.
- `createAuditLog`: logs refreshes.

`getActiveInstruments()` selects all active instruments ordered by asset class and symbol.

`buildMarketOverviewRow(instrument)`:

1. Tries to get a market price for one instrument.
2. If successful, returns symbol, name, asset class, currency, price, last update, source, cache age, and stale status.
3. If failed, returns the row with `marketPrice: null` and `source: "Unavailable"` instead of crashing the whole dashboard.

`getMarketOverview(req, res)`:

1. Loads active instruments.
2. Uses `Promise.all` to fetch price rows for all of them.
3. Logs `MARKET_OVERVIEW_REFRESHED`.
4. Returns the watchlist as JSON.

### `src/controllers/auditLogController.js`

Imports `pool`.

`getAuditLogs(req, res)`:

1. Queries the `audit_logs` table.
2. Renames snake_case DB columns into camelCase JSON fields.
3. Orders newest first.
4. Limits to 50 rows.
5. Returns logs as JSON.
6. Returns HTTP 500 if the query fails.

### `src/controllers/tradeController.js`

This is the largest controller because it owns trade capture, trade refresh, detail lookup, and reporting.

Imports:

- `pool`: database access.
- `calculatePnL`: P&L formula.
- `validateTrade`: input validation.
- `getLatestMarketPrice`: market data service.
- `createAuditLog`: audit events.

`formatDateForTradeId(dateValue)`:

1. Converts the given trade date into a JavaScript `Date`.
2. If invalid, uses today's date.
3. Formats it as `YYYYMMDD`.
4. This becomes the date part of `TRD-YYYYMMDD-000001`.

`generateTradeId(tradeDate)`:

1. Calls PostgreSQL sequence `trade_id_sequence`.
2. Pads the number to 6 digits.
3. Returns a trade ID like `TRD-20260621-000006`.

`isActiveInstrument(symbol)`:

1. Queries `instruments`.
2. Requires matching symbol and `is_active = TRUE`.
3. Returns true/false.

`refreshBookedTradesMarketPrices()`:

1. Selects all trades with `status = 'BOOKED'`.
2. Uses a `Map` called `pricesByInstrument` so each instrument price is fetched once.
3. For each trade, gets the latest/cached price.
4. Recalculates P&L.
5. Updates `market_price`, `pnl`, `last_price_updated_at`, and `market_data_source`.
6. If one price refresh fails, it logs a warning and continues.
7. If any trades were recalculated, it creates a `PNL_RECALCULATED` audit log.

`createTrade(req, res)`:

1. Reads `instrument`, `tradeType`, `quantity`, `tradePrice`, and `tradeDate` from the request body.
2. Normalizes instrument and trade type to uppercase.
3. Calls `validateTrade`.
4. Prepares fallback values for rejected records.
5. Checks the instrument against reference data.
6. If the instrument is invalid/inactive, logs `TRADE_REJECTED` and returns HTTP 400.
7. If validation passed, fetches latest market price.
8. If market data fails, returns HTTP 502.
9. Generates a trade ID.
10. Sets final status to `REJECTED` if validation failed or `BOOKED` if validation passed.
11. Calculates P&L only for accepted trades.
12. Inserts the trade using parameterized SQL.
13. Logs `TRADE_CREATED`.
14. Logs `TRADE_REJECTED` for rejected trades.
15. Logs `TRADE_VALIDATED` and `TRADE_BOOKED` for booked trades.
16. Returns JSON with message, trade ID, status, market price, P&L, and rejection reason.
17. Handles duplicate trade ID with PostgreSQL error code `23505`.
18. Handles other failures with HTTP 500.

Important interview point: parameterized SQL prevents SQL injection because user values are passed separately from the SQL text.

`getTrades(req, res)`:

1. Refreshes market prices and P&L for booked trades.
2. Selects all trades.
3. Aliases DB column names into frontend-friendly names like `TradeId`.
4. Orders newest first.
5. Returns rows as JSON.

`getTradeByTradeId(req, res)`:

1. Reads `tradeId` from the URL.
2. Looks up one trade by `trade_id`.
3. Returns 404 if not found.
4. Returns the full trade record if found.

`getTradeReport(req, res)`:

1. Counts total trades.
2. Counts booked trades.
3. Also returns `ValidTrades` for backward compatibility.
4. Counts rejected trades.
5. Sums P&L only for booked trades.
6. Returns one summary object.

## Service Files

Services hold reusable business logic.

### `src/services/validationService.js`

`validateTrade(trade)` returns a rejection reason string or `null`.

Current rules:

1. Instrument is required.
2. Trade type must be `BUY` or `SELL`.
3. Quantity must be greater than 0 and numeric after conversion.
4. Trade price must be greater than 0 and numeric after conversion.
5. Trade date is required.

The function returns the first failed rule. Returning one clear reason makes rejection messages easy to display and audit.

Note: this validation is simple and uses JavaScript `Number(...)`. It rejects obvious bad values like `abc`, `NaN`, `0`, and negative values. A stricter version can separately reject empty strings/null before conversion.

### `src/services/pnlService.js`

`calculatePnL(tradeType, quantity, tradePrice, marketPrice)`:

1. Normalizes trade type to uppercase.
2. Converts quantity, trade price, and market price to numbers.
3. Defines a helper that rounds to 4 decimals.
4. For `BUY`, returns `(marketPrice - tradePrice) * quantity`.
5. For `SELL`, returns `(tradePrice - marketPrice) * quantity`.
6. For invalid trade type, returns 0.

Why formulas differ:

- A BUY makes money when market price rises.
- A SELL makes money when market price falls.

### `src/services/marketDataService.js`

This file integrates with Twelve Data and implements server-side cache.

`require("dotenv").config()` loads market data provider settings.

Constants:

- `DEFAULT_CACHE_TTL_MS = 60 * 1000`: 60 seconds.
- `EXTENDED_CACHE_TTL_MS = 120 * 1000`: 120 seconds.
- `priceCache = {}`: in-memory cache object.
- `EXTENDED_TTL_SYMBOLS`: FX and commodity symbols that use 120 seconds.

`getProvider()` reads `MARKET_DATA_PROVIDER` and defaults to `twelvedata`.

`getApiKey()` reads `MARKET_DATA_API_KEY`.

`normalizeProviderName(provider)` accepts variants like `twelve_data` and normalizes them to `twelvedata`.

`buildTwelveDataUrl(symbol, apiKey)` creates the Twelve Data URL safely using `URL` and `searchParams`.

`getCacheTtl(symbol)` chooses 120 seconds for FX/gold symbols and 60 seconds for others.

`getCachedPrice(cacheKey, allowStale = false)`:

1. Looks up a cached value.
2. Returns `null` if there is no cache entry.
3. Checks whether it expired.
4. If expired and stale values are not allowed, returns `null`.
5. Otherwise returns price data with `fromCache: true`, `stale`, and `cacheAgeSeconds`.

`setCachedPrice(cacheKey, symbol, data)` stores a fresh price with expiration metadata.

`fetchFromTwelveData(symbol, apiKey)`:

1. Calls the external API.
2. Parses JSON safely.
3. Throws an error if provider fails.
4. Converts returned price to number.
5. Rejects invalid or non-positive prices.
6. Returns normalized market data.

`getLatestMarketPrice(symbol)`:

1. Normalizes symbol.
2. Reads provider and API key.
3. Validates symbol and API key.
4. Builds a cache key.
5. Returns valid cache if available.
6. Fetches from Twelve Data if cache is missing/expired.
7. If API fails, returns stale cache if available.
8. Throws if no fresh or stale data exists.
9. Stores fresh data in cache.
10. Returns normalized market data.

### `src/services/auditLogService.js`

`createAuditLog(eventType, entityType, entityId, description)` inserts one row into `audit_logs`.

It uses parameterized SQL:

```sql
VALUES ($1, $2, $3, $4)
```

That prevents SQL injection.

It catches errors internally and logs a warning. This is intentional because audit logging should not break trade booking or market refresh.

## `database.sql`

The first comment explains that the database should be created before running the script.

`CREATE TABLE IF NOT EXISTS instruments (...)` creates the reference data table.

Instrument columns:

- `id`: primary key.
- `symbol`: unique market symbol.
- `name`: display name.
- `asset_class`: FX, Equity, Commodity, Crypto.
- `currency`: quote/reporting currency.
- `is_active`: controls whether users can trade it.
- `created_at`: timestamp.

The `INSERT INTO instruments ... ON CONFLICT DO NOTHING` block seeds sample instruments and avoids duplicate insert errors.

`CREATE SEQUENCE IF NOT EXISTS trade_id_sequence START 1;` creates the sequence used for generated trade IDs.

`CREATE TABLE IF NOT EXISTS trades (...)` creates the trade table.

Trade columns:

- `id`: internal primary key.
- `trade_id`: user-facing unique trade ID.
- `instrument`: symbol.
- `trade_type`: BUY or SELL.
- `quantity`: trade size.
- `trade_price`: user-entered trade price.
- `market_price`: latest market price used for booking/P&L.
- `pnl`: calculated profit/loss.
- `trade_date`: transaction date.
- `status`: BOOKED or REJECTED final lifecycle status.
- `rejection_reason`: reason for rejected trades.
- `last_price_updated_at`: timestamp of latest market price refresh.
- `market_data_source`: source like `twelvedata`.
- `created_at`: insert timestamp.

`CREATE TABLE IF NOT EXISTS audit_logs (...)` creates the audit trail table.

Audit columns:

- `id`: primary key.
- `event_type`: event name like `TRADE_BOOKED`.
- `entity_type`: entity category like `TRADE`.
- `entity_id`: trade ID or symbol.
- `description`: readable explanation.
- `created_at`: event time.

`ALTER TABLE trades ADD COLUMN IF NOT EXISTS ...` makes the script safe for older databases missing newer columns.

`DELETE FROM trades; DELETE FROM audit_logs;` clears demo data when reseeding.

`ALTER SEQUENCE trade_id_sequence RESTART WITH 1;` resets generated trade IDs.

The `INSERT INTO trades (...) VALUES ...` block seeds four booked trades and one rejected trade.

`SELECT setval('trade_id_sequence', 5, TRUE);` sets the sequence so the next new trade becomes number 6.

The final `INSERT INTO audit_logs` block seeds audit events for demo trades.
